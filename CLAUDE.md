Default to using Bun instead of Node.js.

- Use `bun <file>` instead of `node <file>` or `ts-node <file>`
- Use `bun test` instead of `jest` or `vitest`
- Use `bun build <file.html|file.ts|file.css>` instead of `webpack` or `esbuild`
- Use `bun install` instead of `npm install` or `yarn install` or `pnpm install`
- Use `bun run <script>` instead of `npm run <script>` or `yarn run <script>` or `pnpm run <script>`
- Bun automatically loads .env, so don't use dotenv.

## APIs

- `Bun.serve()` supports WebSockets, HTTPS, and routes. Don't use `express`.
- `bun:sqlite` for SQLite. Don't use `better-sqlite3`.
- `Bun.redis` for Redis. Don't use `ioredis`.
- `Bun.sql` for Postgres. Don't use `pg` or `postgres.js`.
- `WebSocket` is built-in. Don't use `ws`.
- Prefer `Bun.file` over `node:fs`'s readFile/writeFile
- Bun.$`ls` instead of execa.

## Testing

Use `bun test` to run tests.

```ts#index.test.ts
import { test, expect } from "bun:test";

test("hello world", () => {
  expect(1).toBe(1);
});
```

## Frontend

Use HTML imports with `Bun.serve()`. Don't use `vite`. HTML imports fully support React, CSS, Tailwind.

Server:

```ts#index.ts
import index from "./index.html"

Bun.serve({
  routes: {
    "/": index,
    "/api/users/:id": {
      GET: (req) => {
        return new Response(JSON.stringify({ id: req.params.id }));
      },
    },
  },
  // optional websocket support
  websocket: {
    open: (ws) => {
      ws.send("Hello, world!");
    },
    message: (ws, message) => {
      ws.send(message);
    },
    close: (ws) => {
      // handle close
    }
  },
  development: {
    hmr: true,
    console: true,
  }
})
```

HTML files can import .tsx, .jsx or .js files directly and Bun's bundler will transpile & bundle automatically. `<link>` tags can point to stylesheets and Bun's CSS bundler will bundle.

```html#index.html
<html>
  <body>
    <h1>Hello, world!</h1>
    <script type="module" src="./frontend.tsx"></script>
  </body>
</html>
```

With the following `frontend.tsx`:

```tsx#frontend.tsx
import React from "react";

// import .css files directly and it works
import './index.css';

import { createRoot } from "react-dom/client";

const root = createRoot(document.body);

export default function Frontend() {
  return <h1>Hello, world!</h1>;
}

root.render(<Frontend />);
```

Then, run index.ts

```sh
bun --hot ./index.ts
```

For more information, read the Bun API docs in `node_modules/bun-types/docs/**.md`.

## CLI Development Best Practices

This project implements a CLI tool with the following coding standards:

### Output and Logging Standards
- **Use `process.stdout.write()` for data output** - Never use `console.log()` for data that users need to capture
- **Use structured logging with pino** - All informational, debug, and error messages should use the logger
- **Separate data from logs** - Query results go to stdout, execution info goes to stderr via logger
- **Support verbose/quiet modes** - Use `-v/--verbose` for detailed logging, quiet by default for script usage

### Type Safety Requirements
- **Avoid `any` types** - Use `unknown` for error handling and cast to `Record<string, unknown>` when needed
- **Proper error handling** - Always handle both Error instances and unknown error types
- **Explicit type annotations** - Define interfaces for all API request/response structures

### CLI Architecture Patterns
- **Use Commander.js** - Structured argument parsing with proper help text and validation
- **Auto-detection with fallback** - Detect common values (like project ID) but allow explicit override
- **Multiple authentication methods** - Support various auth patterns with proper validation
- **Format flexibility** - Support both human-readable and machine-readable output formats

### Code Quality Standards
- **Run `bun run lint` before commits** - Ensure biome formatting and TypeScript checks pass
- **Structured error responses** - Consistent error format across all error conditions
- **Comprehensive help text** - Include examples and authentication method documentation
